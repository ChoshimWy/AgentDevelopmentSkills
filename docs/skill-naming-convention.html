1. 核心决策

Canonical Apple entry：apple-orchestration。本版本直接移除旧 runtime/vendor 命名，不保留 alias、shim 或重复 Skill。
  • 共享编排使用 workflow-orchestration；平台 overlay 使用 <platform-id>-orchestration。
  • 平台 ID 必须来自 platforms/<id>/manifest.json。
  • Skill 名称处于扁平安装命名空间，目录名、frontmatter name 与 Binding target 必须一致。

2. 命名语法

^[a-z0-9]+(?:-[a-z0-9]+)*$
maximum length: 64
folder name == SKILL.md frontmatter name

名称应简短、稳定、描述职责；平台 overlay 优先使用稳定作用域加职责。

3. 作用域规则

类型格式示例说明
跨平台 Discipline<domain>-<role>workflow-orchestration、code-review不得伪装成平台入口。
平台级能力<platform-id>-<role>apple-orchestration覆盖平台 Manifest 全部 targets 时使用。
目标系统能力<target>-<role>ios-automation能力确实收窄到目标系统时使用。
工具链能力<toolchain>-<role>xcode-build工具链构成明确触发边界时使用。
生态产品能力<ecosystem>-<role>app-store-changelog生态名比平台前缀更准确时使用。

禁止 phantom Skill：预留名称不表示能力已实现。只有安装真实 Provider、Binding、权限与 Conformance 后才能创建 canonical entry。

5. 禁止项与受控例外

  • 所有新 Skill 禁止使用 codex-、claude-、gemini-、openai- runtime/vendor 前缀。
  • 禁止把 subagent、并发数量或当前模型写成稳定平台入口。
  • 例外只能由 policy 显式声明；当前 Apple package 的 deprecated 与 grandfathered 均为空。

Apple 当前统一使用 apple-orchestration、apple-debugging、apple-design-context-compiler 与 apple-verification；真实只覆盖 iOS 的能力继续使用 ios-。

6. 重命名规则

  1. 在同一 change set 中移动 Skill 目录，并同步 frontmatter name、Provider Binding、Taxonomy、Agent prompt、当前架构文档与 lint。
  2. 默认直接移除旧名称，不创建 alias、shim 或重复可安装副本；确需兼容时必须由用户或发布策略明确要求，并在 policy 中声明生命周期。
  3. 不可变迁移来源、迁移映射的 source_path 与 legacy comparison fixtures 保留历史名称；所有当前 target path 与 package inventory 使用 canonical 名称。
  4. 重命名完成后运行命名门禁、Skill Schema lint、Migration Audit 和完整 Conformance。

7. 新增或重命名门禁

python3 scripts/validate_skill_naming.py
python3 platforms/apple/scripts/lint_skill_schema.py --skills-dir <package>/skills
python3 scripts/run_conformance.py

新增平台时先更新平台 Manifest 与 skill-naming-policy.json。bootstrap-only 平台携带 Skill、implemented 平台缺少 canonical orchestration、Binding 指向旧名称都会失败。

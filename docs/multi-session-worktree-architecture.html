1. 结论与边界

可以抽离，而且应当抽离。大部分生命周期合同与具体平台无关：Worktree 创建/清理、稳定 Commit Base、Dirty Gate、Session Registry、Patch Hash、依赖图、Review Scope、证据新鲜度与 Integration Gate。平台包只拥有构建环境指纹、缓存槽位、验证 Adapter 和依赖工具链规则。

共享 Core / Discipline

定义“一个 Session 是什么、从哪里创建、改了什么、证据是否仍有效、何时允许集成”。不出现 Xcode、Gradle、npm、Docker 等命令。

平台 Overlay

定义“如何构建和验证、哪些缓存可共享、如何计算环境兼容性、哪些生成目录必须隔离”。

首期选择：先实现通用 Git Workspace + Workflow Session 合同和 Apple Binding。其他平台仍为 bootstrap-only；文档中的候选策略不是已安装能力，也不能生成 ready Plan。

2. 问题与目标

当前风险

  • 多个 Session 共用一个 Working Tree 和 Index，修改、暂存区、Review 范围及清理动作互相污染。
  • 从 dirty 工作区隐式派生任务时，没有不可变 Base SHA，验证与审查证据无法稳定绑定。
  • 简单地“每 Session 一份完整缓存”虽然隔离，但会让 Apple 等大型工程重复编译。

目标

  • 源码、Index、Branch、Diff 与 Session 生命周期隔离。
  • 构建缓存按“平台环境兼容性”共享，不按 Session ID 或 Worktree Path 粗暴切分。
  • Review 和验证绑定同一 repository identity set 与聚合 source_identity，任一相关仓库修改都会使旧证据失效。
  • 跨平台合同可复用，平台实现保持 Manifest、权限、版本与 Conformance 独立。
非目标：不复制当前 dirty 目录、不自动应用 diff/stash、不让 Core 选择 workspace/scheme/device/Gradle task/package-manager，也不在本阶段实现远程 Session 编排服务。

3. 分层架构

User / Agent CLI
  └─ agent-session create | inspect | checkpoint | gate | finish
       ├─ Shared Git Workspace Contract
       │    ├─ worktree + branch + immutable base
       │    └─ repository patch fingerprint + changed paths
       ├─ Shared Workflow / Review Contract
       │    ├─ session registry + dependency graph
       │    ├─ evidence freshness + Session Gate
       │    ├─ Integration Gate
       │    └─ Final Delivery Gate
       └─ Selected Platform Binding
            ├─ Apple: Xcode + build-queue + Shared DerivedData
            ├─ Android: Gradle strategy (future Provider)
            ├─ Web: package/build cache strategy (future Provider)
            └─ Backend/Desktop: runtime/build strategy (future Provider)

通用层只依赖 Capability ID。CLI 名称使用 agent-session，不把共享生命周期固化为 ios-session；Apple 通过已选平台 Binding 补充验证和缓存行为。

5. Session 数据模型

运行时元数据建议由 Workflow owner 保存到主仓 $(git rev-parse --git-common-dir)/agent-sessions/<session-id>.json。它是本机协调状态，不提交到业务分支；团队级默认策略可另存为仓库内版本化配置。一个 Session 可以是纯 Git、单平台或多平台，也可以包含主仓及本地依赖仓的多个隔离 Worktree。

6. 生命周期与 Gate

clean integration checkpoint C0
  ├─ create Session A @ C0
  └─ create Session B @ C0

Session: created → active → checkpointed(clean commit set)
                            → verified/reviewed → gated → integrated → closed
                                               ↘ blocked

Session Gate
  = every repository HEAD == frozen checkpoint commit
  = every Session/dependency Worktree is clean
  = current repository identity set == verification repository identity set
  = current repository identity set == review repository identity set
  = platform evidence satisfies requested strength
  = Adapter/Ledger identity and artifact hashes are valid
  = reviewer actor is independent from implementation actors

Integration Gate
  = accepted Session commits merged
  + overlap/shared-boundary review
  + integration verification

Delivery Gate
  = target branch baseline → final delivery branch review/verification
  • 普通并行：多个 Session 从同一 Integration Commit 创建。
  • Stacked：下游只能基于上游显式 Checkpoint Commit，并由 Workflow Registry 登记依赖；不能读取上游未提交文件。
  • 重叠修改：单 Session Review 仍独立，但重叠 hunk 标记为 shared-boundary 并强制 Integration Review。
  • 完成：先为每个可写仓生成最终 Checkpoint Commit，要求所有 Session/dependency Worktree clean，再针对该不可变 commit set 执行最终验证与 Review。Gate 通过后只允许集成这些精确 Commit SHA，不得再生成或改写 Commit；清理 Worktree 是最后一步。

7. 三层指纹模型

指纹Owner决定什么必须包含不得包含
Repository Patch共享 Git单个仓库 Change Set 身份；Session identity 是排序后的仓库 identity 集合repository ID、算法版本、Base SHA、tracked binary diff、staged/unstaged、排序后的 untracked 路径/类型/内容Session ID、绝对 Worktree 路径、临时日志
Environment平台共享哪个构建缓存槽工具链、SDK/运行时、项目标识、依赖锁、关键构建配置Branch、Session ID、完整源码 Diff
Verification平台验证验证证据能否复用Environment、Base、Patch、验证强度、测试选择、目标只凭 artifact 路径或最新时间

Patch 算法必须冻结 Git 配置影响：禁用 external diff/textconv，使用 binary/full-index 输出，路径按原始字节稳定排序，并明确 symlink、submodule、文件模式和超大 untracked 文件策略。首期无法安全处理的对象必须 fail-closed，不能静默漏算。多仓 Session 对每个仓分别计算 Repository Patch，再以稳定 repository ID 排序生成 Session source identity。开发中身份可覆盖未提交 Patch；Final Gate 必须改用 base commit → checkpoint commit 的 committed source identity，并要求 Worktree clean。

证据合同复用

Worktree 只新增 source identity 上下文，不新建平行的“通过”协议。验证与 Review 继续通过仓库现有 adapter-request-v1、adapter-result-v1、node-attempt-v1 和 run-ledger-v1 记录 request/invocation/plan/node/provider/binding identity、artifact hash、failure attribution 与 cleanup。Gate 还必须保留 structured validation evidence、no_test_reason 规则、blocking findings 以及 reviewer/implementer actor 独立性。

8. Apple / iOS 首期方案

复用现有基础：不重写 build-queue daemon。请求的执行 cwd 切到 Session Worktree；Shared DerivedData 仍按 Apple Environment Fingerprint 分槽并串行写入。

Apple Environment Fingerprint

  • Xcode build/version、Swift language mode、SDK 与 destination platform。
  • workspace/project identity、scheme、configuration、test plan。
  • Podfile.lock、Pods/Manifest.lock、Package.resolved、xcconfig 与关键 Build Settings 的 hash。
  • 不包含 Session ID、Worktree Path、Branch 或 Session source identity。

build-queue 请求增量

{
  "session_id": "device-control-20260715-001",
  "project_id": "sidus-link-pro",
  "worktree_path": "/.../.agent-worktrees/device-control",
  "git_common_dir": "/.../SidusLinkPro/.git",
  "source_identity": "session-source:8c01...",
  "repositories": [{
    "repository_id": "app",
    "role": "primary",
    "worktree_path": "/.../.agent-worktrees/device-control",
    "git_common_dir": "/.../SidusLinkPro/.git",
    "base_commit": "abc123def456...",
    "checkpoint_commit": "789abc012def...",
    "checkpoint_tree": "012def345abc...",
    "patch_hash": "def456789..."
  }],
  "environment_fingerprint": "env-a31f9c...",
  "derived_data_slot": "sidus-link-pro/env-<sha256(environment-fingerprint)>",
  "destination": "platform=iOS Simulator,id=...",
  "test_plan": "AcruxTests",
  "target_fingerprints": ["target:app-tests"],
  "mode": "checkpoint"
}

daemon 必须校验

  • Worktree 真实存在、不是不受信任 symlink，且解析后的 Git Common Dir 与注册值相同。
  • Session 已注册，当前完整 repository identity set 与请求的 source_identity 相同；TOCTOU 时在获取构建锁后再次计算。
  • DerivedData lease Slot 必须等于 project-id / env-sha256(environment-fingerprint)；它只是 system DerivedData root 上的协调身份，不是客户端可提交的路径或独立 DerivedData 目录。
  • build-check Artifact 根目录由 daemon 映射到 queue-owned artifacts / session-id / source-identity / attempt-id / job-id；完成时冻结相对路径、mode、size 与 SHA-256 manifest，cache reuse 必须同时重验 manifest bytes 和目录当前 inventory。

build-for-testing / test-without-building 产物租约

Shared DerivedData Slot 只是缓存，不是可复用验证证据。同一槽位可被不同 Session source identity 顺序写入，因此从 build-for-testing 到 test-without-building 的完整验证尝试必须满足下列二选一合同：

  1. 持有同一 Environment Slot 的原子写租约直到测试结束；或
  2. 把 .xctestrun、test bundle 与所需产物固化到不可变 attempt artifact，并在执行前校验 environment/source/target/test-bundle/test-plan/destination fingerprints 与 SHA-256。

锁释放后若 Shared DerivedData 被另一 Patch 写入，旧 Session 不得仅凭原路径执行 test-without-building。身份或租约无法证明时必须重新 build-for-testing；结构化结果同时记录实际执行 bundle hash，防止“证据绑定 A、二进制来自 B”。

当前实现边界：daemon lease 只覆盖单个 queue job，因此 Worktree Session 的 test-without-building 当前明确 fail-closed；只有后续把 immutable build artifact identity 作为 daemon 输入并完成执行前复验后才可解锁。普通非 Worktree request 的历史 action 不受此门禁影响。

身份保证分级：多仓 committed source identity 由 daemon 在 pre/locked/post 三个边界重算，destination 与 test plan 绑定实际命令路径；当前 request 中的 environment/target fingerprints 仍是 caller-frozen assertion，尚未由 daemon 重新发现 Build Settings 或重算 target source。结构化摘要必须标记该 assurance，不能仅凭字段存在宣称环境/目标身份已验证。

Apple 依赖边界

内容策略
DerivedData / Module Cache按环境槽共享；同槽写入继续走 shared build-queue 锁。
CocoaPods 下载缓存可共享下载缓存；每个 Worktree 独立生成 Pods/、lock 与 workspace，禁止软链共享可写 Pods。
本地 :path Pod不得让多个 Session 指向同一个可写组件目录。组件仓必须绑定只读稳定 Commit，或作为当前 Session 的 dependency Worktree；其 repository base/patch identity 同时进入 Session、请求和证据。不修改 Pods/ 快照，不自动切线上依赖。
SwiftPM共享下载缓存；每个 Worktree 独立持有 Package.resolved，锁文件变化切 Environment Slot。本地 package 与本地 Pod 使用同一多仓 identity 规则。
生成文件优先输出到 Derived/Temp 目录；项目内生成物必须纳入所属 Repository Patch。

其他外部本地 Framework、Unity 工程或生成源码仓同样不得作为未建模的共享可写输入。无法解析稳定 Commit/content hash 或隔离 dependency Worktree 时，Apple 验证请求 fail-closed。

所有 Xcode 验证仍统一通过项目根 codex_verify.sh 或 ~/.codex/bin/codex_verify 接入 shared build-queue；不得因 Worktree 改造改走裸 xcodebuild 或 Xcode MCP。

10. 建议接口

共享 CLI

agent-session create <name> --base <commit-or-ref> [--platform <id>]...
agent-session list [--json]
agent-session inspect <session> [--refresh]
agent-session checkpoint <session> --message <subject>
agent-session gate <session>
agent-session finish <session>
agent-session remove <session> --require-closed

若调用方没有显式 --base，只有当前 Worktree 干净且 policy 允许推断时才可使用 HEAD；dirty 时必须拒绝并提示指定 Commit/Ref。显式 Base 存在时可以从 dirty 调用目录创建，但必须说明本地修改不会继承。

平台 Hook 合同

resolve_environment(session_context) -> environment_identity
resolve_cache_slot(environment_identity) -> opaque_slot_id
plan_verification(session_context, environment_identity) -> request
execute_verification(request) -> structured_evidence
validate_evidence(session_identity, evidence) -> fresh | stale | invalid

这些 Hook 必须由现有 Adapter request/result 与 Ledger attempt 承载。通用层只校验 Hook 输入/输出 Schema、权限、actor 和 identity 一致性，不解析平台内部字段，也不直接调用平台命令。

11. Fail-closed 规则

禁止：从 dirty 工作区隐式创建、把 Index/Stash 当 Base、复制另一个 Session 未提交内容、验证不绑定 Patch、Review 全仓未知 Diff、共享无锁可写缓存、客户端注入任意缓存路径、未声明 Stacked 依赖、未过 Gate 集成。
  • 每个仓库 Base 必须解析为完整 Commit SHA；分支名只作为来源记录，不能代替已冻结 SHA。
  • 所有路径先 canonicalize，再验证位于允许根；拒绝越界、替换后的 symlink 和不匹配的 Git Common Dir。
  • Registry 更新使用 lock + 临时文件 + fsync/atomic replace；未知 Schema、非法状态转换和重复 Session ID 拒绝。
  • 所有 Repository Patch 与聚合 Session source identity 在 Review/验证开始前和 Gate 裁决时重算；Final Gate 只接受 clean Worktree 的 committed source identity，任一不一致即 stale。
  • 本地依赖若未绑定只读 Commit/content hash 或独立 dependency Worktree，拒绝创建验证请求。
  • Adapter/Ledger identity、artifact hash、actor 独立性或 blocking/no-test 字段缺失时，拒绝把结果视为 passed。
  • Gate 后不得创建、amend、rebase 或由 hook/filter 改写 Commit；Integration 只能接收 Gate 冻结的精确 Commit SHA 集合。
  • Integration Worktree 必须干净；任何未归属修改都阻塞新 Session 默认创建与集成。
  • Session 清理不得自动丢弃修改；Worktree/Branch 删除需要 closed 状态或显式人工接管。

14. 风险与回退

风险控制回退/阻塞
通用层吸收平台细节Capability + opaque platform envelope；Core 搜索 Xcode/Gradle/npm 等词移回平台包后再合并
Patch Hash 漏文件或受 Git 配置影响冻结算法、禁 external diff、覆盖 untracked/symlink/mode、golden fixtures未知对象 fail-closed；升级算法版本
共享 DerivedData 缓存污染环境分槽、daemon 二次校验、验证尝试租约或 immutable bundle hash、结构化诊断身份不明时重新 build-for-testing；必要时切隔离槽或清槽
共享可写本地依赖越过 Patch identity多仓 Session、dependency Worktree、pinned commit/content hash无法冻结依赖即阻塞验证
Session evidence 绕过 Adapter/LedgerContext 只作强类型 envelope；Gate 重验 invocation/provider/artifact/actor identity缺任一身份即 invalid
Worktree 路径/Registry TOCTOUcanonical path、Git Common Dir 绑定、原子 Registry、执行前重验拒绝请求并保留诊断
Integration 自动合并破坏人工修改Integration clean gate、显式策略、冲突即停止保留 Session Worktree，交人工处理
未来平台被文档提前宣称 readyManifest implementation status 与 Conformance 双门禁继续 bootstrap-only
回退原则：Apple overlay 可关闭并退回现有单 Worktree codex_verify 路径；不得通过复制 dirty 内容或绕过 shared build-queue 来“临时兼容”。通用 Session 元数据保留为只读诊断，Worktree 由人工确认后清理。

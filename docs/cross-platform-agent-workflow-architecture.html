1. 结论先行

推荐方案：单一仓库 + 独立平台包 + 按平台选择安装 AgentDevelopmentSkills 统一承载 Core、Apple、Android、Web、Backend、Desktop 与 QA/Design 包；各平台在仓内保持独立目录、Manifest、版本、权限和 Conformance 边界，安装器只部署用户选择的平台集合。
Core→Shared Discipline→Platform→Runtime Config→Repository→Module
  • 嵌套 AGENTS.md 提供目录级确定性覆盖；project-router 处理根目录、混合仓库和跨端任务。
  • 本机只允许一个受管全局 ~/.codex/AGENTS.md；平台包不得各自覆盖该路径，平台专属细节优先保留在 Skill/Manifest，少量 always-on 规则通过带 scope 的分片确定性合成。
  • Manifest 自描述平台检测信号和工作流，Core 不硬编码所有平台细节。
  • Desktop 是平台族，QA 是横向 discipline；二者都不是某个移动端 Skill 的附属功能。
  • Product Design 作为 Design discipline 的能力提供者复用；Figma / Sketch 通过独立 Adapter 接入。
  • Core 采用 Discovery → Policy → Plan Compiler → Execution/Evidence 四引擎，而不是由一个编排器承担全部决策。
  • 工作流以稳定 Capability ID 和 DAG 表达，具体 Skill 名称通过插件 Binding 晚绑定。
  • 开发验证与正式 QA 使用两套等级，避免“build 通过”等同于“质量覆盖完成”。
  • iOSAgentSkills 已先以兼容桥冻结行为，再通过 Migration Audit v2 拆分到 platforms/apple/、五个共享 Discipline 与显式 Codex Runtime Config；源码态与隔离安装态不再依赖 sibling 仓。
  • Apple 为 implemented；Android、Web、Backend、Desktop 为 bootstrap-only，发现结果只能受控阻塞，不能生成 phantom Binding 或 ready Plan。

2. 背景与问题

现有 Apple 工作流包含统一主入口、任务分型、CP0–CP3、Fail-Fix-Report、最窄验证、结构化证据、独立 reviewer、quick-verify、wrapper/queue 分层和 Apple 私有依赖边界。验证不再依赖已打开的 Xcode 窗口或 Xcode MCP；当前 wrapper 已统一项目发现、构建、测试、exact-request 去重/成功缓存与证据归一化入口。Verification Session、三层 fingerprint、same-or-stronger 跨请求复用、优先级与自动 .xctestrun 复用仍是独立 scaffold/后续 daemon 集成合同。

其中一部分是跨平台通用合同，另一部分高度依赖 Xcode、Swift、CocoaPods、真机/模拟器与 Apple 工具链。P2C 已把 workflow、review、documentation、git 与 Canonical UI IR 基础抽为独立 Discipline，把 Apple binding 留在平台包，并将全局 Codex 配置拆为显式 Runtime Config；单仓不等于规则平铺到同一入口。

Apple 官方知识源联邦接入platforms/apple/scripts/apple_official_expertise.py 只检查 Xcode 默认受管目录或用户显式核实的非默认本机导出，并输出 source trust、Xcode/SDK 身份、文件 hash、已审查路由与 eligibility 等 metadata；任意本地或第三方目录默认保持 unverified。它不联网、不执行导出脚本、不复制 Skill 正文，也不新增第二套实现入口。已知能力通过稳定 Capability ID 增强 ios-feature-implementation、xcode-build、ios-automation、apple-debugging 与 apple-code-review；unverified/unknown Skill、版本缺失、SDK 不满足或冻结 hash 漂移均 fail-closed。

2.1 问题清单

  • 根目录启动时，单靠 Skill description 难以稳定判断目标模块。
  • Package.swift、package.json 等信号存在歧义。
  • Monorepo 与共享协议修改可能同时影响多个平台。
  • 平台工作流之间会重复 task classification、review、report 等通用合同。
  • QA 被压缩为“跑测试”，缺少策略、用例、兼容、回归和发布判断。
  • Desktop 不能被简化为单个平台或单一框架。

3. 目标、非目标与设计原则

目标

  • 自动识别仓库、模块、平台、栈与测试能力。
  • 按需加载最小工作流集合。
  • 复用 iOS 已验证的编排与证据纪律。
  • 让 QA 支持独立和伴随两种模式。
  • 通过 schema 统一跨平台交付。

非目标

  • 不创建一个万能开发 Skill。
  • 不立即实现所有平台能力。
  • 不把平台命令硬编码到 Core。
  • 不替代项目级 AGENTS.md。
  • 迁移验证完成前不删除 iOSAgentSkills 原入口。

原则

  • 仓库事实优先于语义猜测。
  • 显式目标优先于自动检测。
  • 规则靠近 ownership。
  • 最小验证、结构化证据。
  • 兼容迁移、可回退。

7. Core 四引擎与 Capability Graph

为避免项目检测、规则合并、计划生成和工具执行互相耦合，Core 应拆成四个可独立测试的引擎：

Discovery Engine→Policy Resolver→Plan Compiler→Execution / Evidence Runtime
引擎输入输出硬边界
Discovery Engine仓库、cwd、目标文件、Git 状态、Manifest只读 Project Facts / Candidate Modules不选择工作流，不执行项目脚本
Policy Resolver用户约束、AGENTS、项目规则、Manifest PolicyResolved Policy + Decision Trace低优先级规则不得覆盖锁定约束
Plan CompilerProject Profile、Task Classification、Resolved PolicyCapability DAG + Missing Capabilities不绑定未经安装或不兼容的实现
Execution / Evidence Runtime已冻结 DAG、权限与 Adapter Binding执行状态、结构化证据、最终报告不得伪造未执行节点或静默降级

Capability 优先于 Skill 名称

Core 只依赖稳定能力 ID；插件将能力绑定到当前 Skill、脚本或工具。这样平台包可以重构内部 Skill，而不破坏 Core。

{
  "provider": "ios-agent-skills",
  "provides": [
    "implementation.apple.ui",
    "verification.apple.affected-tests",
    "verification.apple.auto",
    "automation.apple.device",
    "performance.apple.profiling"
  ],
  "bindings": {
    "implementation.apple.ui": "ios-feature-implementation",
    "verification.apple.affected-tests": "apple-verification",
    "verification.apple.auto": "apple-verification",
    "automation.apple.device": "ios-automation"
  }
}
能力选择规则 同一能力存在多个 Provider 时，按用户显式选择、项目本地绑定、兼容版本、平台匹配度和优先级解析；无法唯一选择时进入 ambiguity，不按发现顺序随机绑定。

8. Policy、Decision Trace 与 DAG Workflow Plan

8.1 规则合并语义

除优先级外，每个字段必须声明合并策略：replace、append、union、intersect、deny-wins 或 locked。安全禁止项默认 deny-wins，用户显式约束进入 locked。

{
  "decision": "select verification.apple.affected-tests",
  "reason_code": "TARGET_FILE_AND_MODULE_MATCH",
  "source": "apps/ios/AGENTS.md",
  "confidence": 0.98,
  "merge_strategy": "locked",
  "overridden_candidates": [],
  "evidence": ["apps/ios/App.xcworkspace", "apps/ios/Podfile"]
}

CLI / Agent 表面应支持 route --explain 与 plan --dry-run，让用户看到“为什么选中、为什么排除、缺什么能力”。

8.2 Workflow Plan 使用 DAG

{
  "id": "ios-validation",
  "capability": "verification.apple.auto",
  "requires": ["ios-affected-tests"],
  "mandatory": true,
  "permission_profile": "project-read-execute",
  "status": "pending",
  "fallback": null
}
contract analysis
├─ iOS implementation ─→ iOS validation ─┐
├─ Android contract validation ───────────┤
└─ Backend implementation → API tests ────┤
                                          ↓
                              cross-platform QA
                                          ↓
                                independent review

8.3 缺失、冲突与降级

  • 必需能力缺失：节点与计划为 blocked。
  • 可选能力缺失：计划为 degraded，必须披露覆盖缺口。
  • 插件冲突、循环依赖或不兼容：在执行前 fail-closed。
  • Fallback 必须显式声明且语义等价；禁止静默切换到较弱验证。
  • 每个节点区分 mandatory、optional 与 advisory。

9. 自动识别与路由合同

9.1 证据优先级

用户显式指定›目标文件 / 模块›cwd›Git changes›嵌套规则›Manifest›README 推断

9.2 路由阶段

  1. 查找 Git root 或用户指定 root，不存在 Git 时以 cwd 为扫描边界。
  2. 读取插件 registry 与 Manifest，限制扫描深度和忽略目录。
  3. 收集 strong / medium / weak signals，识别候选模块根。
  4. 结合任务文本、目标路径和 changed files 计算目标模块。
  5. 生成 project profile；冲突进入 ambiguities，不强猜。
  6. 任务分型后生成 workflow plan，并列出未安装平台包。
  7. 主 Agent 在 CP0 锁定计划，再进入实现或 QA。
反例约束 Package.swift 可能是 iOS、macOS、Server Swift 或 CLI；package.json 可能是 frontend、backend、desktop shell 或 tooling。单个中弱信号不得直接定案。

9.3 跨端协议示例

用户任务：修改 packages/api-schema/openapi.yaml

10. 平台 Manifest

平台包通过 Manifest 自描述检测信号、模块标识、工作流、验证适配器和兼容版本。Core 只解释协议，不维护每个平台全部文件规则。

{
  "lock_schema_version": "2.0",
  "id": "apple",
  "kind": "platform",
  "targets": ["ios", "macos", "watchos", "tvos", "visionos"],
  "detection": {
    "strong": ["*.xcworkspace", "*.xcodeproj", "Podfile"],
    "medium": ["Package.swift", "*.entitlements"],
    "weak": ["*.swift", "Info.plist"]
  },
  "module_markers": ["*.xcworkspace", "*.xcodeproj", "Package.swift"],
  "workflows": {
    "default": ["apple-orchestration"],
    "implementation": ["ios-feature-implementation"],
    "verification": ["apple-verification"],
    "automation": ["ios-automation"],
    "build": ["xcode-build"],
    "performance": ["ios-performance"]
  },
  "package": {
    "name": "apple-agent-skills",
    "source": "platforms/apple",
    "compatibility": ">=1.0 <2.0",
    "install_group": "apple"
  },
  "requires": [],
  "optional_requires": ["design.canonical-ir"],
  "conflicts": [],
  "replaces": [],
  "permissions": {
    "detection": "repository-read-only",
    "implementation": "project-write",
    "verification": "project-read-execute"
  }
}

Desktop Manifest 额外声明 target 与 framework hints；QA Manifest 作为 discipline package 声明 test strategy、case design、regression、compatibility 和 reporting，而不是伪装成平台。

11. iOSAgentSkills 整合边界

抽到 Core / Shared Discipline保留在 Apple 包
本地事实优先、最小改动、写入前计划Swift / Structured Concurrency / MainActor 规则
任务分型、lite/standard/full、CP0–CP3Xcode、SDK、availability、Apple 文档
Fail-Fix-Report、独立 reviewer 门禁Verification Session、三层 fingerprint、quick/checkpoint/final lane 与 codex_verify/build-queue
验证等级与结构化证据原则workspace / scheme / device / DerivedData 约束
通用 code-review / reporting 字段CocoaPods、私有 :path Pod、Pods snapshot ownership
Skill schema、Manifest 和安装注册协议SwiftUI/UIKit、iOS 自动化、XCTest、Archive / Export

单仓迁移策略

  1. 先兼容、后迁入：P2A 外部 Manifest 保留为冻结回归基线；P2B 已将 Skills、脚本、配置、规则和 Provider Manifest 迁入本仓 platforms/apple/。
  2. 双路径验证：同一组代表性任务分别走原入口和 Core 路由，比较工作流与门禁。
  3. 单仓去重：P2C 已由 disciplines/workflow、review、documentation、git、design 承担通用合同，platforms/apple/ 只保留 Apple 专属实现与 binding，禁止双份可安装真源。
  4. 配置显式化：Codex profiles/shared config 位于 runtime-configs/codex，必须显式 opt-in；选择 Apple 不得隐式改变全局工具行为。
  5. 迁移期可回退：源码与隔离安装已以本仓为真源；install.sh 只迁移可精确识别的旧 iOSAgentSkills 软链，按当前用户约束不生成持久配置备份，失败时只使用单进程临时快照回滚受管目录并恢复旧软链，临时数据随安装结束清理，未知本地内容不得静默覆盖。

14. 验证、QA 与审查门禁

Verification Level

NONE → STATIC → LINT → AFFECTED_TESTS
→ MODULE_BUILD → INTEGRATION → UI_SMOKE → FULL

确认代码改动的基本正确性，默认选择覆盖风险的最低等级。

QA Coverage Level

SMOKE → TARGETED → REGRESSION
→ COMPATIBILITY → END_TO_END → RELEASE_CANDIDATE

确认产品质量覆盖，不由一次构建或单测通过替代。

实现任务默认收口

Platform implementation→Targeted validation / no_test_reason→Independent code-review→CP3
  • Reviewer 不可用：blocked / pending review，不得由实现者自审。
  • 平台验证工具与环境规则由平台包决定；Core 只规定证据质量与升级条件。
  • 优先结构化报告；不默认读取完整 raw log、trace、result bundle 或缓存目录。
  • 同类失败最多执行 2 次 fail-fix-report 回环，之后显式阻塞。

15. Design / UI·UX 总体架构

19. Capability Contract 与 Runtime State Machine

Capability ID 只能解决“能力叫什么”，还必须用版本化合同定义输入、输出、副作用、权限、幂等性、并发资源和失败语义。两个 Provider 只有满足同一合同，才能被视为可替换实现。

{
  "id": "verification.apple.affected-tests",
  "version": "1.0",
  "input_schema": "affected-test-request-v1",
  "output_schema": "verification-report-v1",
  "permission_profile": "project-read-execute",
  "side_effects": ["build-artifacts"],
  "idempotent": true,
  "concurrency_keys": ["xcode-build-queue"],
  "failure_codes": [
    "tool-unavailable",
    "environment-blocked",
    "validation-failed",
    "contract-violation"
  ]
}

19.1 节点状态机

pending→ready→running→passed / failed / blocked / skipped / cancelled→stale
状态语义允许的后续动作
pending依赖尚未满足依赖完成后进入 ready
ready依赖、权限和资源前置条件已满足获取资源锁后运行
running已冻结输入与 attempt identity完成、失败、阻塞或取消
passed输出 Schema 与 artifact hash 均有效解锁下游节点
failed当前改动或能力执行失败按策略重试或 fail-fix-report
blocked缺少权限、工具、环境、审批或必需能力外部条件变化后 resume
skipped由策略显式跳过且理由已记录仅在 Plan 重编译后恢复
cancelled主动中止并完成清理新 attempt 才能重启
stale输入、Policy、Lockfile、环境或上游证据已变化禁止复用，必须重新执行

19.2 重试、恢复与幂等

  • 每个节点记录 attempt_id、timeout、最大重试次数、首个失败和清理结果。
  • 只有 Capability 声明 idempotent=true 且 fingerprint 未变化时，Runtime 才能自动重试。
  • 中断后 resume 必须重新验证 Lockfile、权限、资源锁、输入 hash 和上游 artifact。
  • 取消必须终止子进程、释放资源锁、恢复临时权限并归档部分证据。
  • 节点通过但报告写入失败时，可从已校验 artifact 重建报告，不重复执行有副作用的操作。

20. Resource Scheduler、Approval Gate 与 Run Ledger

20.1 并发资源模型

DAG 节点只有在依赖满足且资源不冲突时才能并行。资源键必须稳定、可解释并在取消或失败时释放：

repository-write:<repo-id>
git-index:<repo-id>
build-queue:<toolchain-id>
device:<device-id>
simulator:<simulator-id>
design-document-write:<document-id>
deployment:<environment>
database-migration:<database-id>
  • 写集重叠、同一 Git index、同一设备或同一设计文档写入不得并行。
  • 只读探索允许并行，但必须基于相同的冻结 repository snapshot。
  • Reviewer 获取冻结 diff、验证故事与合同，不读取仍在变化的工作树。
  • 资源等待、持有者、超时、取消和释放必须进入 Run Ledger。

20.2 Human Approval Gate

高副作用节点必须暂停计划并请求明确授权，审批范围不可从一个 document、environment 或 repository 静默扩大到另一个。

默认需要审批：安装未知插件、扩大网络/凭据/文件系统权限、写回 Figma/Sketch、发布部署、数据库迁移、生产配置修改、破坏性 Git 操作和外发机密设计资源。

21.1 真源分工

21.2 机密数据与保留策略

  • Figma、Sketch、产品截图、测试数据和设计资产默认不上传未授权外部 Provider。
  • 凭据、访问 Token、用户数据不得进入 Agent Packet、报告或 Run Ledger。
  • 设计数据只保留目标 document/page/node slice；报告优先记录 hash 和受控路径。
  • 截图、日志和缺陷证据支持脱敏；Artifact retention 必须声明保留周期、owner 与自动清理策略。
  • 外部 Provider 执行前披露数据范围、目的和权限；卸载插件时清理其缓存和失效凭据。

21.3 版本、弃用与迁移

  • Core、Manifest、Capability Contract、Schema 和平台包分别使用版本号，并声明兼容范围。
  • 弃用遵守 deprecated → warning → blocked-new-use → removal，不能直接删除仍被 Lockfile 使用的能力。
  • Schema migration 必须可 dry-run、可验证、可回退，并保留迁移前 hash。
  • 插件升级先执行 Conformance Suite 和 Plan diff，用户确认后才更新 Lockfile。

安装器从同一发布物按需部署 Core 与用户选择的平台包，不应把所有平台 Skill 无条件复制到运行目录。平台选择必须可审计、可重复并写入 Lockfile：

  • 源码仓选择安装入口为根目录 install.sh；TTY 默认从所有平台 Manifest 生成复选菜单，Apple/iOS 同时满足 Manifest implemented 与 source-installer activation/smoke handler readiness，因而默认显示为 [x]；Android、Web、Backend、Desktop 以 [ ] 和 bootstrap-only 状态展示但暂不可选；使用 ↑ / ↓ 移动光标、Space 选择或取消、Enter 确认，确认后清除整个平台菜单；非交互模式使用可重复的 --platform 或 --platform all，--json 必须显式选择平台。当前可安装闭包仍为 Apple + Codex Runtime Config，并支持 --dry-run / --target-root；默认人类可读结果直接从所选平台的预览/完成状态开始，不重复显示产品标题与分隔线，并保留变更/验证摘要，目标目录、规划平台、Runtime Config、安装包、Skill 数量及旧软链迁移状态等完整元信息由显式 --json 以 canonical JSON 输出；不联网也不默认执行 pip。dry-run 与真实安装共用 managed-root preflight；除普通文件形式的 macOS Finder 元数据 .DS_Store 外，冲突时不得覆盖用户本地自定义 Skill。
  • 远程发布入口采用两个薄 bootstrap：POSIX install.sh 与 Windows install.ps1 先下载 canonical release manifest，从其版本固定的 asset_base_url 获取共享 bootstrap_install.py，并在执行前校验 bootstrap size/SHA-256；共享 Core 再验证 host artifact size/SHA-256 与 ZIP 规范化、安全解压边界，调用发布物中的同一 scripts/install_local.py。构建真源 scripts/build_release_bundle.py 生成确定性 universal source bundle、bootstrap asset hashes 与 source identity。默认 production manifest 只声明 macOS/Linux（WSL2 视为 Linux）；PowerShell 入口存在不等于 Windows host-ready，未通过 Windows 路径、权限、事务和 installed smoke 前必须因缺少 windows artifact fail-closed。

  • 官方平台包来源固定为发布物内 platforms/<id>/；环境变量或用户 registry 只用于显式授权的本地扩展包。
  • 本机 Xcode 官方 Skill 导出属于运行时知识源，不属于发布物或 package source：Lock/报告只记录 Xcode build、SDK major、导出内容 hash、routing hash 与所采用的规则标识，禁止把导出正文、引用或工具脚本再分发进本仓。
  • --platform 支持单个平台、通过重复参数选择多个平台，以及 all；只有 implemented 平台可安装，bootstrap-only 平台返回明确阻塞；未选择的平台不得产生 Skill、binding 或权限副作用。
  • --runtime-config 必须显式选择；平台依赖闭包不得隐式激活 profiles、全局配置或插件开关。
  • 未安装包可被检测到，但工作流计划必须标记 missing_capabilities。

23.1 单一全局 AGENTS 与平台规则合成

禁止 last-writer-winsApple、Android、Web 等平台包不得分别把自己的根 AGENTS.md 软链或复制到同一个 ~/.codex/AGENTS.md。当前 iOSAgentSkills 的单平台软链方式只能作为迁移输入，不能直接复用于多平台安装器。
  • 唯一全局入口：安装器只生成一份受管全局指令文件，并让 model_instructions_file 指向它；Core 拥有该文件，平台包只能贡献声明式分片。
  • 最小常驻：平台通用安全、路由和 fail-closed 规则进入 Core；Xcode、Gradle、浏览器等详细规则保留在对应 Skill/Manifest，避免多平台安装导致全局上下文膨胀。
  • 显式作用域：必须常驻的分片声明稳定 rule ID、platform/capability scope、优先级与 merge strategy；无 scope、重复 ID、locked 冲突或互斥规则在写入前 fail-closed。
  • 确定性生成：按 Core → dependency-ordered shared discipline → selected platform → explicit runtime config 的固定顺序渲染 canonical 文本；Lock v2 记录每个分片 hash、顺序、rule trace 和最终指令 hash。

  • 可逆卸载：移除某个平台后，从剩余已选包重新生成全局文件，不删除其他平台规则或用户自定义内容。

每份 Workflow Plan 同时生成 fingerprint，包含 project profile、resolved policy、插件锁定、目标模块、环境基线和 DAG identity；只有 fingerprint 相同时才允许复用缓存证据。

事务边界源码安装器覆盖受管根目录的单进程 staged swap；install.sh 将后置 smoke/activation 纳入底层单进程临时回滚窗口，失败时恢复写入前受管目录或旧软链，但按用户约束不创建持久配置备份。跨进程互斥、进程崩溃恢复日志、doctor 自动修复以及 upgrade/uninstall 事务仍属于 Phase 6，当前不得宣称为跨进程原子安装或发布包完成。

23.3 插件权限与信任边界

  • Discovery 默认只读，不执行仓库任意脚本，也不访问凭据。
  • Implementation、Verification、Design Write 使用不同 permission profile。
  • 设计工具默认只读，写入 Figma / Sketch 必须获得用户显式授权并锁定 node scope。
  • 安装记录来源、版本和内容 hash；未知或被篡改的插件 fail-closed。
  • 插件不得以 fallback 为名扩大网络、文件系统、设备或凭据权限。

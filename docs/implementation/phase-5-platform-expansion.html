1. 范围与成功标准

启动前置门禁先完成 Apple/iOS 隔离安装态合同 smoke，并在获准真实 Xcode 工程完成 host-level build/test smoke；在此之前不启动 Android、Web、Backend Provider 实现，也不扩大 Phase 4 Desktop 最小 Adapter 范围。
禁止把占位当实现Android、Web、Backend Manifest 当前仍声明 implementation_status=bootstrap-only，不含 capabilities、bindings 或 installation；Desktop 已由 Phase 4 提供最小 implemented Provider，未冻结的 framework capability 仍必须 blocked/manual，不得伪造支持。
Phase 5 目标把平台差异完全收敛到 Manifest、Capability Contract 与 Adapter；Core 仅使用通用 Project Profile、Policy、DAG、Runtime 和报告协议。

本阶段包含

  • 仓内平台包模板、版本/权限/证据最低合同。
  • Android、Web、Backend 首批 Provider。
  • Compose / Web UI Binding 与可选 Desktop Binding。
  • 共享 API schema、Monorepo 与跨端 QA workflow。

本阶段不包含

  • 不在 Core 硬编码 Gradle、npm、Go、Java 或部署命令。
  • 不承诺覆盖每种框架与 CI Provider。
  • 不默认执行发布、数据库迁移或生产部署。
  • 不因平台缺失静默降低 mandatory 验证。

推荐扩展顺序

Apple installed-contract smoke → 获准真实 Xcode host smoke → Android → Web TypeScript → Backend Node/Java/Go → Desktop 增量 → Unity → Flutter / React Native。每一步必须先通过模板 Conformance，再进入下一平台。

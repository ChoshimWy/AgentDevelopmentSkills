1. 范围与成功标准

Phase 1 目标 给定仓库事实、任务上下文和插件 Manifest，稳定生成 Project Profile、Resolved Policy、Decision Trace 与 Workflow DAG，并通过 fake adapter 验证节点状态、资源锁、恢复和报告合同。

本阶段包含

  • 四引擎 Core 的最小实现。
  • Capability / Policy / DAG / Runtime Schema。
  • JSON Manifest registry 与兼容性校验。
  • 确定性项目识别和安全 unknown。
  • route --explain 与 plan --dry-run。
  • Fake adapters、fixtures、golden tests。

本阶段不包含

  • 不修改或迁移 iOSAgentSkills。
  • 不执行真实 Xcode、Gradle、npm 或设备任务。
  • 不连接 Figma / Sketch / Product Design。
  • 不实现 Android、Web、Backend、Desktop 平台包。
  • 不实现安装器、远端 registry 或发布流程。
  • 不承诺完整生产级分布式调度。

阶段成功标准

  1. 相同输入和 Lockfile 得到 byte-stable canonical JSON 与相同 fingerprint。
  2. 单平台、Monorepo、共享协议、unknown、ambiguous 和缺失能力均得到预期计划。
  3. Policy 决策可以解释来源、合并策略和被覆盖候选。
  4. DAG 无循环，mandatory 能力缺失时 fail-closed。
  5. Runtime 能验证通过、失败、阻塞、取消、恢复和 stale 状态。
  6. 全部核心输出通过版本化 Schema 校验。

技术决策记录Phase 1 采用 Python 3.11+ 与标准库优先，以复用现有工作流脚本经验、降低安装成本，并保持 macOS、Windows、Linux 的一致入口；JSON-only Manifest 避免引入 YAML 解析依赖，所有指纹均基于 UTF-8、键排序、紧凑格式且带末尾换行的 canonical JSON。Node.js/TypeScript 更适合后续 UI 工具链，Rust 更适合高性能执行器，均保留为未来 adapter 或 Runtime 重写边界，不进入 Core 合同。
阻塞条件P02–P05 未冻结前不得开始大规模实现；允许先创建最小 Schema spike，但不得形成不可逆目录或 API。

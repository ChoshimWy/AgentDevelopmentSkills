1. 范围与成功标准

平台真值platforms/desktop 当前没有 Skills、bindings 或 installation assets，只能产生 bootstrap_required；本阶段必须安装真实 Desktop Provider 后才能解锁 ready Plan。
Phase 4 目标QA Core 负责风险、策略、用例、缺陷、回归与报告；平台包负责具体环境与执行 Adapter。Verification Level 与 QA Coverage Level 必须保持正交。

本阶段包含

  • QA plan/case/result/defect/report Schema 与 orchestration。
  • 开发伴随模式和独立 QA 模式。
  • PRD、Bug、Release 三类端到端 QA fixtures。
  • Desktop 检测、环境画像与最小 build/test/UI smoke binding。

本阶段不包含

  • 不让一次 build/test 通过替代 QA 结论。
  • 不要求小任务启动所有 QA subAgent。
  • 不实现完整设备农场或所有 Desktop framework。
  • 不把 Desktop 平台细节写入 QA Core。

阶段成功标准

  1. 每个 QA 流程有入口/退出条件、环境、覆盖、证据与残余风险。
  2. Bug 流程能记录复现、严重度、归因、修复验证与回归 ownership。
  3. Release 流程能输出明确的 go / conditional-go / no-go 建议。
  4. Desktop 最小包可被替换或扩展而无需修改 QA 主合同。

4. S2 PRD、Bug、Release 三类 QA 流程

状态ID任务完成判定
□S2-01PRD 流程：需求切片、可测性、验收标准与风险识别输出 traceable requirement-to-case matrix
□S2-02PRD 流程：边界、异常、权限、离线、兼容与可访问性用例关键状态无无主覆盖缺口

□S2-05Release 流程：变更盘点、风险抽样、兼容矩阵与 known issues覆盖依据和排除理由可解释
□S2-06Release 流程：go / conditional-go / no-go 决策阻塞项、豁免 owner、到期条件和 residual risk 完整
□S2-07三类流程支持 partial / blocked / cancelled不得用“未执行”伪装 passed
